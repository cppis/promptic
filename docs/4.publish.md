## 1. 사전 준비

### 1.1 npm 계정

[https://www.npmjs.com](https://www.npmjs.com) 에서 계정을 생성하고, 터미널에서 로그인합니다.

```bash
npm login
npm whoami   # 로그인 확인
```

### 1.2 패키지 이름 중복 확인

```bash
npm info prompatic   # 이미 등록된 패키지인지 확인
```

패키지가 없으면 `npm error code E404`가 출력됩니다 — 정상입니다.
이미 존재하고 본인 계정 소유가 아니면 `@cppis/prompatic`처럼 scope를 붙여야 합니다.

## 2. npm 인증 설정

npm 계정에 **2FA(2단계 인증)** 가 활성화된 경우, 일반 `npm publish`는 403 에러가 발생합니다.
아래 방법 중 하나를 선택합니다.

### 방법 A — Granular Access Token 생성 (권장)

CI/CD 환경이나 반복 배포에 적합합니다.

1. [https://www.npmjs.com/settings/tokens](https://www.npmjs.com/settings/tokens) 접속
2. **Generate New Token → Granular Access Token** 선택
3. 설정:
   - **Token name**: 예) `prompatic-publish`
   - **Expiration**: 90일 또는 원하는 기간
   - **Packages and scopes**: `Read and write`
   - **Require 2FA**: 비활성화 (배포 자동화가 필요한 경우)
4. 생성된 토큰을 복사하여 로컬 npm 설정에 저장:

```bash
npm config set //registry.npmjs.org/:_authToken <발급받은_토큰>
```

저장 위치는 `~/.npmrc`이며, 이후 `npm publish`를 추가 입력 없이 실행할 수 있습니다.

### 방법 B — OTP 직접 입력

일회성 배포 시 사용합니다. Google Authenticator 등의 앱에서 현재 OTP를 확인하고 전달합니다.

```bash
npm publish --otp=<6자리_OTP>
```

OTP는 30초마다 갱신되므로 명령어 실행 전 최신 값을 확인합니다.

## 3. 배포 전 체크리스트

```bash
cd mcp-server
```

### 3.1 테스트 통과 확인

```bash
npm test
```

전체 테스트가 통과해야 배포를 진행합니다.

### 3.2 버전 확인

`package.json`의 `version`이 npm에 이미 배포된 버전보다 높아야 합니다.

```bash
cat package.json | grep '"version"'
npm info prompatic version   # 현재 배포된 버전 확인
```

### 3.3 배포 포함 파일 확인

`package.json`의 `files` 필드에 배포에 포함될 파일이 올바르게 지정되어 있는지 확인합니다.

```json
"files": [
  "index.js",
  "lib/",
  "package.json",
  "README.md"
]
```

dry-run으로 실제 포함 파일 목록을 미리 확인합니다.

```bash
npm pack --dry-run
```

### 3.4 README 최신화

`README.md`의 버전 표기, 설치 안내, 기능 목록이 현재 코드와 일치하는지 확인합니다.

## 4. 버전 업 및 배포

### 4.1 버전 규칙

[Semantic Versioning](https://semver.org/) 을 따릅니다.

| 변경 종류 | 명령어 | 예시 |
|-----------|--------|------|
| 버그 수정, 테스트 수정 | `npm version patch` | `0.5.1` → `0.5.2` |
| 새 기능 추가 (하위 호환) | `npm version minor` | `0.5.1` → `0.6.0` |
| 하위 호환 불가 변경 | `npm version major` | `0.5.1` → `1.0.0` |

`npm version` 명령은 `package.json` 버전 수정 + git commit + git tag를 자동으로 수행합니다.

```bash
# 예: 패치 버전 업
npm version patch
```

### 4.2 버전을 수동으로 지정하는 경우

자동 git tag가 필요 없거나 버전을 직접 지정하는 경우 `package.json`을 직접 수정합니다.

```bash
# package.json의 "version" 필드를 원하는 값으로 수정 후
git add package.json
git commit -m "chore(release): bump version to 0.5.2"
git tag v0.5.2
```

### 4.3 배포 실행

```bash
# 기본 배포 (Granular Token 설정 완료 시)
npm publish

# 2FA OTP를 직접 입력하는 경우
npm publish --otp=<6자리_OTP>

# scope 패키지 (@cppis/prompatic) 배포 시
npm publish --access public
```

### 4.4 전체 흐름 요약

```bash
cd mcp-server

# 1. 테스트
npm test

# 2. 버전 업 (patch / minor / major 중 선택)
npm version patch

# 3. 배포
npm publish

# 4. git push (태그 포함)
git push && git push --tags
```

## 6. 트러블슈팅

### E403 — 2FA 또는 토큰 권한 부족

```
npm error code E403
npm error 403 Forbidden — Two-factor authentication or granular access token with bypass 2fa enabled is required
```

→ [2. npm 인증 설정](#2-npm-인증-설정)을 참고하여 Granular Token을 생성하거나 `--otp` 플래그를 사용합니다.

### E403 — 버전 중복

```
npm error code E403
npm error 403 You cannot publish over the top of existing version
```

→ `package.json`의 버전을 올린 후 다시 배포합니다.

### E404 — 패키지 미존재 (최초 배포 시)

`npm info` 실행 시 E404가 나오는 것은 정상입니다. `npm publish`를 실행하면 최초 등록됩니다.

### ENEEDAUTH — 로그인 필요

```
npm error code ENEEDAUTH
```

→ `npm login`을 실행하거나, `~/.npmrc`에 토큰이 올바르게 설정되어 있는지 확인합니다.
